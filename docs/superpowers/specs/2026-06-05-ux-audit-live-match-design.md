# Stakeholder UX Audit & Live-Match Workflow Optimisation

**Date:** 2026-06-05
**Issue:** [#61](https://github.com/bennylim0926/BES/issues/61)
**Branch:** `feat/issue-61-ux-audit`

## Summary

Make the BES app fully self-service for all 5 roles (Admin, Organiser, Judge, Emcee, Helper) so users can operate independently without verbal instruction or hand-holding. The Emcee becomes the primary live-match operator with a simplified BattleControl interface, a broadcast timer, and auto-unlock voting at 10s remaining. Every role gets clear empty states, actionable error recovery, next-step nudges, and role-aware guidance.

---

## Roles

Five roles, two auth pathways:

| Role | Auth | Token-based |
|------|------|-------------|
| Admin | Username/password | No |
| Organiser | Username/password | No |
| Judge | Token redemption | Yes |
| Emcee | Token redemption | Yes |
| Helper | Token redemption | Yes |

Token-based roles are "session roles" — generated by Admin/Organiser, redeemed at `/auth/token`, locked to a specific event.

## Stream A: BattleControl Restructure

### A.1 Backend Authorization

Currently `BattleController` has class-level `@PreAuthorize("hasAnyRole('ADMIN', 'ORGANISER')")`. Emcee has zero backend battle access despite frontend router allowing `/battle/control`.

Replace class-level annotation with method-level annotations:

**Operator tier** (Admin, Organiser, Emcee):
- `GET /battle-pair` — read current pair
- `GET /phase` — read current phase
- `GET /judges` — read judge list
- `GET /bracket` — read bracket state
- `GET /state` — full state snapshot
- `GET /active-genre` — read active genre
- `GET /champions` — read champions
- `GET /smoke` — read smoke list
- `GET /overlay-config` — read overlay config
- `POST /active-genre` — switch genre
- `POST /phase` — set phase (LOCKED → VOTING, etc.)
- `POST /score` — calculate score from votes
- `POST /revote` — reset votes

**Config tier** (Admin, Organiser only):
- `POST /battle-pair` — set pair
- `POST /bracket` — set bracket state
- `POST /judge`, `DELETE /judge` — manage judges
- `POST /judge/weightage` — judge weightage
- `POST /overlay-config` — overlay settings
- `POST /champion-reveal` — champion reveal
- `POST /battle-mode` — battle mode
- `POST /smoke` — set smoke list
- All `DELETE` endpoints

No new database migration needed — this is a code-only change.

### A.3 New Components

**BracketViewer.vue** (read-only, both roles)
- Props: `rounds`, `topSize`, `currentRound`
- Displays bracket match cards without drag handles or edit controls
- Highlights: current match (amber glow), winners (green), upcoming (dimmed)
- Designed for easy reading — larger names, clear round labels, no clutter

**BracketEditor.vue** (interactive, Admin/Organiser only, inside Setup)
- Props: `rounds`, `topSize`, `unplacedParticipants`, `currentRound`
- Events: `update-rounds`, `start-from-match`, `clear-slot`
- Drag-drop from seeding pool into slots, win/clear buttons
- Same visual layout as BracketViewer but with edit affordances

**LiveMatchPanel.vue** (both roles)
- Props: `isReadonly` (true for Emcee)
- Contains all live-match UI: timer, bracket viewer, judge votes, phase actions
- Phase action buttons gated by `isReadonly` — Emcee can advance phase, cannot configure

### A.4 What Emcee CANNOT Do

Enforced at both frontend (component hidden/read-only) and backend (`@PreAuthorize`):

| Action | Frontend | Backend |
|--------|----------|---------|
| Seed bracket / set pairs | Setup hidden | Blocked |
| Add/remove judges | Setup hidden | Blocked |
| Change format (Top 8→16) | Setup hidden | Blocked |
| Change overlay colors | Setup hidden | Blocked |
| Reset bracket | Setup hidden | Blocked |
| Upload images | Setup hidden | Blocked |
| Edit bracket slots | BracketViewer is read-only | Blocked |

## Stream B: Broadcast Timer

### B.1 Timer State Machine

```
IDLE ──[Start]──▶ RUNNING ──[10s left]──▶ UNLOCK → auto-POST VOTING
                    │
                    └──[Reset]──▶ IDLE
```

- **IDLE:** Timer not running. Button shows "Start Timer".
- **RUNNING:** Timer counting down. Button shows "Voting opens in 0:42…" or similar live countdown.
- **At 10s remaining:** Auto-unlock fires. System POSTs `{phase: "VOTING"}` to backend. Timer continues counting 10…9…8…→0. Overlay plays Netflix zoom transition.
- **Reset:** Emcee can reset timer at any point (returns to IDLE).

### B.2 Emcee Freedom

The timer is an assistant, not a constraint. Emcee can:
- Start timer manually at any point
- Reset timer at any point
- **Manually open voting at any point** — the "Open Voting" button is always accessible alongside the timer

### B.3 Component: BattleTimer.vue

- Location: Inside `LiveMatchPanel.vue`
- Presets: 30s, 45s, 60s, 90s (same as audition timer)
- Display: Large countdown number + progress bar
- At 10s: auto-unlock triggers, button shows "Unlocking in 10… 9… 8…" pulsing red
- Manual override: "Open Voting Now" button always available beside timer

### B.4 WebSocket Broadcast

```
BattleTimer (LiveMatchPanel)
  │  STOMP publish to /app/battle/timer
  │  { running: boolean, timeLeft: number, totalDuration: number }
  ▼
Backend @MessageMapping("/battle/timer")
  │  SimpMessagingTemplate → /topic/battle/timer
  ▼
BattleOverlay subscribes /topic/battle/timer
  │  Renders timer bar + countdown
  │  At timeLeft=0: Netflix zoom animation
```

Timer state is **ephemeral** — not persisted to DB. If emcee refreshes, timer resets to IDLE.

### B.5 Overlay Timer Display

**Placement:** Thin progress bar centered at top of overlay, 60% width. Does not overlap battler names, images, or VS badge.

```
┌──────────────────────────────────────────┐
│  ┌─────────────── 0:42 ───────────────┐  │  ← progress bar + countdown
│  │  ████████████████░░░░░░░░░░░░░░░░  │  │
│  └────────────────────────────────────┘  │
│                                          │
│   ┌──────────┐    VS    ┌──────────┐    │
│   │  LEFT    │          │  RIGHT   │    │
│   │  NAME    │          │  NAME    │    │
│   └──────────┘          └──────────┘    │
└──────────────────────────────────────────┘
```

**Enter animation:** Bar slides down from top + fades in (300ms, `cubic-bezier`)

**Netflix zoom at 0:** The progress bar + number scales from `scale(1)` → `scale(20)` over 400ms with `cubic-bezier(0.55, 0, 0.45, 1)` — rushes past the camera, then cuts to voting indicator. Consistent with the existing overlay's judge panel slam animation but forward rather than lateral.

### B.6 7-to-Smoke Timer

Same per-round timer (LOCKED only, manual start, auto-unlock at 10s). Smoke rounds are faster-paced so shorter presets (30s default) are expected, but the timer component and flow are identical.

**Format-level timer** (overall time limit for 7-to-Smoke, highest-score-wins fallback, tie-breaker handling) is out of scope — tracked as a separate GitHub issue.

## Stream C: UX Guidance Pass

### C.1 Reusable Patterns

All patterns follow the CLAUDE.md design system (Anton SC typography, parallelogram shapes, semantic state left-border + glowing dot, section rules, accent-color tokens).

#### Pattern 1: Empty States

```
┌─────────────────────────────────────────┐
│                                         │
│              (section icon)             │
│                                         │
│           NO PARTICIPANTS YET            │
│     Import from Google Sheets or add     │
│         a walk-in to get started         │
│                                         │
│     ┌─────────────┐  ┌──────────────┐   │
│     │ IMPORT SHEET │  │ ADD WALK-IN  │   │
│     └─────────────┘  └──────────────┘   │
└─────────────────────────────────────────┘
```

Every list/table that can be empty gets a branded empty state with what the section is for and the action to populate it. Never a blank table.

#### Pattern 2: Error Recovery Banners

```
┌──────────────────────────────────────────┐
│ ⚠ Failed to submit score                 │
│   The server returned an error.           │
│   Your score is saved locally.            │
│                    ┌──────────┐           │
│                    │ RETRY NOW │           │
│                    └──────────┘           │
└──────────────────────────────────────────┘
```

Semantic state pattern (3px left border + glowing dot):
- **Red:** Action failed, needs attention, shows retry
- **Amber:** Warning, action may have partially succeeded
- **Green:** Success confirmation, auto-dismisses

Every error must include: what happened, what to do, a retry button where applicable.

#### Pattern 3: Next-Step Nudges

After completing a key action, inline banner shows what comes next:

| After | Nudge |
|-------|-------|
| Creating event | "Next: Add genres to your event →" |
| Importing participants | "Next: Assign judges →" |
| All judges assigned | "Event ready. Go to Audition List →" |
| Releasing results | "Results are now live at /results →" |
| Seeding bracket | "Ready. Switch to Live Match to begin →" |

Inline banners, not modals — don't block, just guide.

#### Pattern 4: Role-Aware Guidance

Every view knows the user's role and tailors messaging:

| Role | Example |
|------|---------|
| Emcee in BattleControl | "You control the battle flow. Start the timer when battlers are on stage." |
| Judge in AuditionList | "Swipe through cards. Tap score to submit." |
| Helper in EventDetails | "Use the Event Day tab for walk-in registration and check-in." |
| Organiser with no event | "Create or select an event to get started." |

### C.3 Error Recovery — Per-Role Specific Cases

#### Judge Errors

| Situation | Target Behavior |
|-----------|-----------------|
| Score submit fails (network) | Red banner: "Score not saved. Tap to retry." Score held locally |
| Voting while phase is not VOTING | Clear message: "Waiting for operator to open voting" |
| Already scored this participant | Green confirmation: "Score submitted ✓" with score shown |

#### Emcee Errors

| Situation | Target Behavior |
|-----------|-----------------|
| Phase advance fails (backend error) | Amber banner: "Could not advance phase. Retry." |
| Timer accidentally started | Manual "Reset Timer" button next to timer |
| Genre switch fails | Banner with retry, genre stays on previous |
| WebSocket disconnects | Amber dot next to phase badge: "Reconnecting…" |

#### Organiser Errors

| Situation | Target Behavior |
|-----------|-----------------|
| Google Sheets import fails | Error with sheet name + row details |
| Participant email send fails | Per-participant error indicator, batch retry |
| Bracket seed with insufficient participants | Amber banner: "Need X more participants to fill Top Y bracket" |

#### Helper Errors

| Situation | Target Behavior |
|-----------|-----------------|
| Walk-in add fails (duplicate) | Red banner: "A participant with this name already exists." |
| Walk-in add fails (missing field) | Red banner highlighting which field is missing |
| Check-in toggle fails | Amber: "Check-in not saved. Tap to retry." |

## Design Constraints

- **Do NOT change** `BattleOverlay.vue`, `BracketVisualization.vue`, or `Chart.vue` — they have their own bespoke CSS and are the reference point, not the target (per CLAUDE.md)
- **BattleJudge.vue** is excluded from the cinematic design system — keep its existing look
- All new UI follows the CLAUDE.md design system: Anton SC typography, parallelogram shapes, semantic state pattern (3px border + glowing dot), section rules, accent-color tokens
- No database migrations needed for this work (timer is ephemeral, auth is code-only)
- Timer is ephemeral — survives page reload only through WebSocket re-sync, no DB persistence

## Out of Scope

- 7-to-Smoke format-level timer (overall time limit) — separate GitHub issue
- Overlay configuration changes beyond timer display
